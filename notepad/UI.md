  - 目前餅乾數（例如:10000,100003）
  - 每秒餅乾產量（cookies per second, CPS）

  - 玩家可以連點
  - 點擊時顯示數字往上跳的特效


右邊（建築物列表區）：
- 建築物列表（每一列是一種建築，可以點擊購買）
  - 顯示：
    - 建築物名稱（例如 Grandma, Factory）
    - 擁有數量（目前已購買幾個）
    - 當前價格（依照規則動態計算後顯示）
  - 互動：
    - 點擊整列嘗試購買該建築
    - 購買失敗時（餅乾不足）顯示錯誤訊息（例如：not enough cookies）





### 畫面目的:「這房子是要給誰住？主要活動是什麼？」只寫：玩家在這個畫面要完成的 1～3 件事。不要寫版面、不要寫公式、不要寫動畫細節。
固定句型（你可以每次都套）：
這個畫面是給【誰】在【什麼情境】使用
使用者在這裡要完成：
A
B
C


這頁面是遊戲的主要遊玩區域,是玩家要在主要遊玩時使用
1. 透過點擊來獲得餅乾
2. 購買建築物來自動生產餅乾
3. 即時看到自己的資源變化,例如餅乾總數,和建築物數量


### 版面結構:「平面圖長怎樣？有哪些房間？」只寫：切幾塊、每塊叫什麼、每塊扮演什麼角色。不要寫「顯示哪些欄位」、不要寫「排序」、不要寫公式。
版面切成：A / B / C（例如：左欄、右欄）
A：一句話角色（例如：主要操作）
B：一句話角色（例如：購買與資訊）

### 互動行為整理:「這房間有哪些開關？按了會發生什麼？」只寫：使用者做什麼動作 → 期望結果（從玩家角度）。不要寫數學公式、不要寫“怎麼算”（那是邏輯層）。
固定句型（業界超常用）：
當使用者【做動作】時，畫面應【立即顯示什麼變化】
若【條件不成立】，則【顯示錯誤/不變動】
例如：
當玩家點擊大餅乾時：餅乾總數立即上升
若購買建築餅乾不足：顯示「餅乾不足」，且數值不改變


點擊大餅乾:
  當點擊餅乾後,總餅乾應該增加對應餅乾數量
購買建築物:(按照價格來排序)
  當點擊某一列建築物:
    如果可以買,則顯示增加後建築物的數量
    如果不能買(錢不夠),顯示"餅乾數量不足"


畫面目的:就是這個頁面(這房間)的主要用途
版面結構:就是大概的格局規劃,像是那一邊是工作區,那一邊是睡覺區
區塊說明:就是詳細的東西有什麼,要放哪,像是我工作區的窗戶旁邊要放伸縮電腦桌
互動行為:就是說站在使用者的立場,做什麼動作ui會怎樣變化,例如我回到我的房間燈就自動開啟
