### 每一節到底要記錄什麼？（費曼版：每節只回答一個問題）

下面我用「你在蓋房子」的比喻來講，並對照你的 `UI.md` 會出現的常見混淆。

#### 1) 畫面目的（Purpose）＝「這房子是要給誰住？主要活動是什麼？」
**只寫：玩家在這個畫面要完成的 1～3 件事。**  
不要寫版面、不要寫公式、不要寫動畫細節。

你寫得不錯（L36–40）。唯一可調整的是：用一句話＋條列就夠了。

**固定句型（你可以每次都套）**：

- 這個畫面是給【誰】在【什麼情境】使用  
- 使用者在這裡要完成：
  - A
  - B
  - C

#### 2) 版面結構（Layout）＝「平面圖長怎樣？有哪些房間？」
**只寫：切幾塊、每塊叫什麼、每塊扮演什麼角色。**  
不要寫「顯示哪些欄位」、不要寫「排序」、不要寫公式。

你在 L45–46 其實已經很好了。  
但像「並且顯示每個建築物擁有數量」這種，算是下一節（顯示資料）才寫會更乾淨。

**固定句型**：

- 版面切成：A / B / C（例如：左欄、右欄）
- A：一句話角色（例如：主要操作）
- B：一句話角色（例如：購買與資訊）

#### 3) 區塊內容（Display / Content）＝「每個房間裡有哪些家具？」
**只寫：看得到的資訊有哪些。**  
可以寫「顯示餅乾總數、CPS、建築名稱、價格、數量」。  
**不要寫：點了會怎樣（那是互動），不要寫公式（那是規則/邏輯），不要寫排序（那是行為/規則）。**

你這節混進了三種東西：

- **互動**：L55「可以點擊並根據現在的數值來增加」→ 這是互動/行為
- **動畫特效**：L56「跳出 +2 +3 特效」→ 這是 UI polish（可以放在「狀態/回饋」或「非 MVP」）
- **公式**：L63 價錢公式 → 這應該在「規則/備註」或 Room B 的設計，不應放在 UI 區塊內容

**固定句型**（每個區塊都這樣寫）：

- 區塊：XXX  
  - 顯示：
    - 資訊 1
    - 資訊 2
    - 資訊 3

#### 4) 互動行為（Interactions）＝「這房間有哪些開關？按了會發生什麼？」
**只寫：使用者做什麼動作 → 期望結果（從玩家角度）。**  
不要寫數學公式、不要寫“怎麼算”（那是邏輯層）。

你這節（L68–75）寫得方向正確，但可以更「可測試」：每一條都用「When / Then」格式，會很像專業 spec。

**固定句型（業界超常用）**：

- 當使用者【做動作】時，畫面應【立即顯示什麼變化】  
- 若【條件不成立】，則【顯示錯誤/不變動】

例如：

- 當玩家點擊大餅乾時：餅乾總數立即上升  
- 若購買建築餅乾不足：顯示「餅乾不足」，且數值不改變

### 你問「有沒有業界通用格式？」我的建議

最通用、最好用、你也最容易模仿的就是這個順序（你已經有了）：

1. **Purpose（目的）**：回答「為什麼存在」  
2. **Layout（版面）**：回答「切幾塊」  
3. **Display（顯示）**：回答「看到什麼」  
4. **Interactions（互動）**：回答「點哪裡、會怎樣」  
（可加）5. **States/Errors（狀態/錯誤）**：回答「錢不夠/空資料怎麼辦」  
（可加）6. **Out of scope（非 MVP）**：回答「以後再做什麼」

這其實就是很多 PRD / UI spec 的骨架，只是不同公司命名不同。

### 你接下來怎麼練到「不用抄也會寫」？

你每寫一節，問自己一句話就好：

- **Purpose**：我在回答「玩家來這裡要幹嘛」嗎？
- **Layout**：我在回答「畫面切幾塊」嗎？
- **Display**：我在回答「看得到哪些資料」嗎？
- **Interactions**：我在回答「做什麼動作→看到什麼結果」嗎？

如果答案不是，那句話就放錯節了。
